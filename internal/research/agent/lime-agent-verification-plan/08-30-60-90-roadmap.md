# 30 / 60 / 90 天路线图

> 目标：在预算有限的前提下，把 Lime Agent 验证体系从“骨架可用”推进到“日常可用”。

## 30 天：低成本 P0 可信化

目标：

```text
不靠 full qcloop 消耗大量 token，也能逐个关闭 P0 blocker。
```

交付物：

- P0 单场景运行手册
- qcloop worker preflight 固化
- GUI owner gate 默认化
- 8 个 P0 的 structured evidence summary 示例
- Claw streaming / cancel 的 runtime transcript 回归
- Design Canvas 保存状态可观察证据

完成标准：

- 每个 P0 至少有单场景 sidecar pass 或明确 blocker。
- 不再出现 worker “命令跑过但 verifier 看不懂”的输出。

## 60 天：Verification Contract 进入开发流程

目标：

```text
每个 Agent 主链改动都有预算标签、Happy Path、Evidence Layers、必跑命令。
```

交付物：

- `internal/exec-plans` 模板
- diff -> P0/P1 场景选择规则
- Supervisor rubric 模板
- Evidence layer 选择规则
- 失败回写规则

完成标准：

- Agent / runtime / Plugin / Skill / Managed Objective 改动不再靠口头判断验证范围。
- 普通改动默认 C0/C1，只有高风险才升级 C2/C3/C4。

## 90 天：差异验证和发布门禁收口

目标：

```text
高风险 Agent 行为可以比较 baseline / candidate，发布只接受 official evidence。
```

交付物：

- flag differential harness 最小版
- benchmark plan / compare 与 Agent QC evidence 对齐
- Managed Objective 纳入 Agent QC 场景
- P0/P1 场景选择规则可由 diff 类型直接推荐
- release summary 消费 official `.lime/qc/agent-qc-evidence.json`
- `agent-qc:audit` 成为发布前完成度审计入口

完成标准：

- release 候选能明确回答：哪些场景 pass，哪些缺证据，哪些是成本/环境 blocker。
- 失败能自动沉淀为更低成本 regression。
