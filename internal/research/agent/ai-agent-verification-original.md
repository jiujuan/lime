# AI Agent 开发，如何省掉那 90% 的手动验证

> 状态：source 整理稿
> 更新时间：2026-07-02

哈喽大家好。

我统计了一下我最近做 AI Agent 开发的时间，结果发现了一件不太对劲的事情。

我真正花在写代码、做设计上的时间，只有 10%。剩下的 90%，全都耗在了同一件事情上：手动测试。

我得点开网页，输入想要测的数据集，盯着 Agent 一步一步地跑，然后用人脑判断这次它到底是变好了，还是变坏了。

而且这还不是最麻烦的。

最麻烦的是，AI Agent 这个系统本身就不确定。有的时候我只是想改一个很小的行为，结果改了一句 prompt，它就崩掉了。

更让人没底的是，很多时候我连它到底有没有真的坏，都没办法一眼看出来。我只能再跑一遍、再跑一遍、再跑一遍，然后凭感觉判断它到底是不是对的。

到这一步，我的工作其实已经不是在写代码，而是在反复手动验证一个连我自己都有点看不懂的系统。

这正常吗？

有没有一种办法，能够把这 90% 的时间，变成一套可重复、不需要我手在屏幕前一遍一遍点的自动化体系？

今天我们就来聊一聊，到底如何让 AI Agent 开发省去这 90% 的时间。

## 这不是一个人的痛点

在聊我们怎么改进它之前，我首先确认了一件事情：这不是我一个人的痛点。

原文里提到，2025 年 OpenAI 的研究员 Jason Wei 提出了一个概念，叫“验证的不对称性”。

意思是，有很多任务做出来很难，但是验证它做得对不对却很容易。

顺着这个，他给出了一条定律，叫 Verifier's Law：

**任何可解且易于验证的任务，终将被 AI 解决。**

这件事情跟我的 90% 有什么关系？

我之所以 90% 的时间都在验证，恰恰是因为对于我们构建的 AI Agent 来说，生成变得很便宜，但是验证还很贵。这就是那个不对称。

也就是说，如果我能把“判断这个 Agent 做得对不对”这件事情，从一个靠肉眼、靠感觉的难题，变成一道自动、可重复的验证体系，那我其实就是把一个难验证的问题，亲手改造成一个容易验证的问题。

一旦它变得容易验证，按照这条定律，剩下的就应该交给 AI 了。

我的角色就变成了：如何制造这个可验证的架构。

## 芯片行业早就走过这条路

这个话听起来很抽象，但其实在另一个行业，三十年前就做过同样的事情。

这个行业就是芯片设计与验证。

写芯片代码的人，往往比验证芯片代码的人要少。芯片真正流片之前，花在验证上的人力经常是设计的两倍、三倍，甚至更多。

为什么会这样？

因为芯片一旦流片，如果出错，那就是几百万美金直接打水漂。没有“改一个 bug 重新部署一下”这种好事。

所以整个行业被现实倒逼着，把大部分精力压在同一件事情上：想尽一切办法，把所有可能的 bug 全部找出来。

干这件事情的核心，不是说你会写芯片代码就行，而是你要会设计验证体系。

你要知道怎么设计一套回归测试，怎么主动把 bug 暴露出来，在哪些地方埋检查点，又怎么衡量自己到底测得够不够全。

讲到这儿你大概已经发现了，这跟我们今天做 AI Agent 开发几乎是同样的形状。

当 AI 把写代码变得越来越便宜，就像当年的 EDA 工具让写 Verilog 变得容易一样，你的价值就不仅仅在设计上面了。

新的价值点是：你能不能设计出一套体系，让 AI 的每一次改动都可以被验证。

## 第一步不是写测试

那么我具体是怎么基于我们自己的 AI Agent 把这件事情做出来的呢？

第一步可能跟你想的不太一样。

我要做的第一件事不是写测试，而是把我们的 AI Agent 改造成一个对别的 AI Agent 友好的系统。

我手上其实有两套系统：一个是我们自己写的 AI Agent，另一套是我用来写代码的 coding 工具，比如 Codex。

我之前测试为什么那么痛苦？

因为那套系统的入口是网页，是 UI，是给人用的。

要测它就得有人在那里点一下、输一下。

从第一天起，它就是为人的眼睛和手设计的。

现在大家都在讲“面向 AI Agent 的开发”，这件事情的真正核心在于三个问题。

第一，你的系统在脱离人的眼睛和 UI 之后，还能跑得起来吗？

第二，它运行过程中的中间状态，有没有留下足够的信息，让另外一个 AI Agent 能看懂？

第三，你有没有专门设计接口，让 AI 能够自主拿到这些信息？

举个具体例子。

我们在运行一个 AI Agent 的时候，我需要让 Codex 能够看到整条 workflow 长什么样，中间调用了哪些工具，每一步拿回了什么数据。

如果这些信息一旦离开 UI 就消失，或者整个 workflow 离开 UI 就停止，后端就瘫了，那只能说明我们的接口到现在为止还是为人设计的，不是为 AI 设计的。

## 前后端分离，是为了让 Agent 能看懂系统

怎么改？

其实有两条比较现成的路。

一条是调用 MCP，模拟网页操作，模拟人的操作，让 AI Agent 直接去独立的网页里靠多模态观察。

另一条是把前后端彻底拆开。

后端变成一个能独立运行的 AI Agent，所有调用都做成命令行控制形式。前端退到一边，只负责把后端的信息监控出来给人看，让人进行操作。

我选的是第二条路：前后端分离。

原因是 MCP 的确能看到网页上的一些操作，但在 authorization、corner case 和效率上，没有前后端分离直接、快速。

做前后端分离这一步，我们也花了不少力气。

原因在于我们之前重度使用 Vercel AI SDK。最开始用的时候确实很好用，但在前后端分离这件事情上，需要花一些功夫，才能做到比较好的程度。

我们把它变成了一个能脱离前端、自己在后台长时间稳定跑下去的系统。

做完这一步，再回过头来看，有没有人跟我们踩了一样的坑？

普林斯顿的 SWE-agent 论文里提到过一个概念，叫 Agent Computer Interface，ACI。

他们的观点是：AI Agent 是一类全新的用户。它和人不一样，所以你得专门给它设计接口。

而且实验结果发现，接口设计得好不好，对 Agent 表现的影响，甚至大过你换一个更强的模型。

所以我们每天听到面向 AI Agent 的编程、面向 AI Agent 的服务，它的核心从来不是某个具体技术。

真正的核心是，你做出来的东西，能不能被另一个 AI，或者更简单一点，能不能被 Codex 快速、主动地拿到它需要的信息。

## 测试不是概念，问题是谁来当裁判

做完前面那步，接下来就很直接了：开始构建测试。

说穿了，它就是一套回归测试，Regression Test。

概念不复杂，复杂的是怎样为一个不确定的系统构建它。

比如我想给 AI Agent 新加一个工具，我心里其实有明确预期：什么条件下这个工具会被触发，触发之后整个 AI 工作流应该怎么走。

这个我期待它怎么走的路径，我们把它叫做 Happy Path。

它就是一个新功能的验收标准。

从架构上来说，我们先要解决一个棘手问题：谁来当裁判？

Agent 的输出是一大段自然语言，是一条运行路径，不是一个可以直接相减的数字。所以传统那种“过”或者“不过”的分类系统是不够用的。

我的办法是把裁判拆成两层：能确定的交给确定性裁判，模糊的交给 AI 裁判。

第一层是确定性的断言，Assertion。

它是一组硬性检查，比如跑测试时工具是否触发，跑到某一步时中间状态有多少条记录，覆盖到了哪些情况。

这一层的特点是绝对可靠、确定性、零成本。

第二层是 LLM 裁判，我把它叫做 Supervisor。

它专门处理第一层卡不住的模糊部分。

这里有几个关键细节。

第一，大语言模型的上下文必须是干净的。

它不参与任何开发，不知道代码来龙去脉，它眼里只有“正确的预期”。

第二，面对那些没有标准答案的输出，我们不让它判断对错，而是让它做量化打分，通过分数判断最终结果好了多少、差了多少。

如果你做过芯片验证，会觉得这套系统很眼熟。

比如我们借用了 Assertion 的概念，借用了 Coverage 的概念，借用了 Scoreboard 的概念。

只不过相比硬件，AI Agent 本身存在模糊性和不确定性，所以它对不确定性的容忍度要更高。

## 攒案例，做开关，然后闭环

解决了谁来当裁判的问题之后，第二步就是不停地攒案例。

上面我们做的是基于 Happy Path 的开发。功能越多，就慢慢把这些 Happy Path 一条一条固化下来。

除此之外，我们还可以从真实用户和测试里，捞那些自己根本没有设计过的五花八门的输入，把它们放进测试集。

这两类回归测试能够保证系统在不停更改的状态下依旧没有退步，同时尽量没有盲区。

还有一个特别关键的点：每加一个新功能，我都会给它配一个开关，一个 flag。

有了开关，就有了开启和关闭两个版本。

当它们跑同一套回归测试时，我就能清楚看到新功能到底带来了什么，哪些变好了，哪些被悄悄弄坏了。

有了这套架构之后，最妙的一步来了：如何把这一切形成闭环？

我们可以在 Codex 里做这样一件事：当它需要开发新功能时，它不仅仅考虑代码怎么写，而是先设计回归测试，其次设计功能开关，然后再编写代码。

写完之后，在开关开启和关闭时，分别跑同一套回归测试。根据测试结果反馈，再修改代码，直到最终代码符合回归测试要求。

在这套体系里，人是不参与具体循环的。

Codex 这种 Coding Agent 通过我们设计的可验证体系，把一切连成闭环。

到这一步你会发现，我们已经不再只是写代码，也不再手动测试。

我们做的是把“什么算对”定义清楚，剩下的就让这个闭环自己收敛。

这正好对应开头那条定律：如果你能验证一个解，就等于给 AI 造出了一个环境，让它去趋近正确答案。

那 90% 的手动验证，到这里就真的变成自动化了。

## 真正要思考的是效率瓶颈

最后聊一下。

今天我们做的前后端分离、回归测试，其实都是抛砖引玉。

我不是想让大家照着我的方案做，而是想让大家看见，为什么我们会做出这样的技术决定。

像前后端分离、回归测试，在很多技术博文里都有。

为什么今天还要反复提？

因为我想让大家看到，如果不是那 90% 的效率浪费把我逼到墙角，我可能不会做这样的技术决定。

这套解法真正想让大家思考的是：在你自己的工程里，有没有类似的瓶颈？有没有正在悄悄吃掉你的时间，或者让你觉得烦躁、重复的问题？

一年前大家还在争，要不要用 AI 写代码。

今天已经没人争了，因为大家都在用。

这个时候真正要思考的是：你怎么使用 AI，才能比别人效率更高？

如果你原来 10 分钟能做完的事情，用 AI 反而干了两个小时，结果还一样，那这不是效率提升，而是本末倒置。

所以不要把我这套解法当成放之四海而皆准的东西。

它可能对你一文不值。

我想给的只是一种启发：看完这个视频之后，回头看看自己的工程里有没有类似的问题。

如果你脑子里有了新的解法，我就很开心。

祝大家都能把属于你的那 90% 效率重新拿回到自己手里。
