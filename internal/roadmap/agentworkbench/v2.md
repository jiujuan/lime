# Agent Workbench v2 可执行协议内核

> 状态：active
> 创建时间：2026-06-12
> 更新时间：2026-06-13
> 范围：`@limecloud/agent-ui-contracts`、`@limecloud/agent-runtime-projection`、`@limecloud/agent-runtime-client`、`lime-rs/crates/app-server`、`lime-agent-workbench` 文档站。

## 结论

v1 已经把 Lime AgentUI 标准从概念页推进到四包 seed、projection、React surface、Content Studio 接入和 Plugin smoke。v2 的目标不是继续补概念，而是补上 AG-UI 真正有价值的工程内核：**事件先经过可执行状态机验证，再进入投影和 UI**。

上一轮分析（Claude Code）方向基本正确，但停留在「清单式对比」，遗漏了两边架构形态的本质差异，也没有实施任何校验内核。v2 在重新精读两边核心管道后重写，以 Lime current 事实源为准，只吸收 AG-UI 的机制，不照搬 AG-UI 的事件名、Observable API 或 legacy compatibility 层。

## 重新精读后的本质差距

### 差距一（最大、上一轮漏掉）：Lime 曾是「全量重投影」，AG-UI 是「流式状态机 + 增量 apply」

v2.1 前，Lime 的 `createAgentUiProjector.apply()`（`packages/agent-runtime-projection/src/uiState.ts`）每来一个事件，就把历史全部事件重新跑一遍 `projectAgentUiState`（重建 messages / timeline / graph / runtimeStatus）。这是 O(N²)，且**没有任何流式合法性约束**——坏流照样能投影出「看起来正常」的 state。v2.1 已把 `apply()` 改为增量 projector，`hydrate()` / `projectAgentUiState()` 仍保留全量路径作为批量基准。

AG-UI 的 `runAgent`（`sdks/typescript/packages/client/src/agent/agent.ts`）是一条真正的流水线：`run -> transformChunks -> verifyEvents -> apply -> subscribers`。其中 `verifyEvents`（`verify/verify.ts`）是一个**有状态闭包状态机**，事件流过时实时维护 `activeToolCalls` / `activeMessages`，非法序列**当场短路**，坏流根本到不了 apply。

### 差距二：Lime 只有「逐事件 schema 校验」，没有「跨事件状态机」

Lime 的 `validation.ts` 全部是逐事件字段校验（某字段是否 string、是否带 scope id、payload 是否过大 / 含密钥）。唯一的跨事件检查是 `collectSequenceIssues`（sequence 是否连续）。文档 `contracts/runtime-event.md` 里写的配对规则（`tool.started -> tool.result/failed`、`action.required -> action.resolved`、`turn.completed` 前收口）**在代码里一行都不存在**，只能靠运行时提供方自觉遵守，违规时只能靠 projection 测试事后兜底。

### 不该当成「缺失」的部分（修正上一轮判断）

- `snapshot.updated` + hydration / repair cursor 是 Lime **比 AG-UI 更完整的恢复模型**，不是「缺少 JSON Patch」。字段级 patch 在 v2.2 先落 schema seed，v2.5 已进入 projection / read model apply 链。
- conformance 等级、fixture 矩阵、event 家族 taxonomy 比 AG-UI 文档更细，是 Lime 的资产，v2 要强化而不是另起炉灶。

## v2 主目标

把 Workbench 从「标准文档 + fixture replay」推进到「协议边界能自己拦截坏流」的工程标准：

```text
App Server / RuntimeCore / ExecutionBackend
  -> RuntimeEvent stream
  -> sequence verifier   <-- v2.0 新增的可执行内核
  -> optional adapter / middleware
  -> projection apply
  -> AgentUI read model / React surface
```

v2 完成后，运行时提供方不能只靠文档自觉遵守配对 / 收口 / 终态规则；这些规则必须可以在 contracts / conformance runner / runtime client 边界**机械执行**。

## AG-UI 深层参考的取舍

| AG-UI 机制 | Lime v2 取舍 | 原因 |
| --- | --- | --- |
| `verifyEvents` 流式状态机 | **吸收**，落到 `@limecloud/agent-ui-contracts` 的 sequence verifier。 | 这是协议能否自证的核心，也是本轮唯一的实施重点。 |
| `transformChunks` | **v2.4 吸收为 runtime event adapter / middleware pipeline**，不照搬 AG-UI Observable。 | Lime 保持现有 JSON-RPC / session gateway surface，只把兼容转换集中到 verifier 前。 |
| `apply` + subscribers Observable 管道 | **吸收核心机制**：runtime client 分发前先过 verifier，projector `apply()` 走增量 accumulator。 | 不照搬 AG-UI Observable API，但保留「先验证、再增量 apply、再通知 UI」的工程形态。 |
| Middleware / backward compatibility | **v2.4 已落最小接口**：adapter / middleware / schemaVersion compatibility middleware。 | 兼容逻辑集中在 runtime client pipeline，不再散落到 projector 或产品 gateway。 |
| JSON Patch `STATE_DELTA` | **v2.2 已落 schema seed，v2.5 已落 projection apply**。 | Lime 现有 `snapshot.updated` / hydration / repair cursor 已成主链，字段级 patch 先以 RFC 6902 语义进入 contracts / JSON Schema，再由 projection/read model apply 消费。 |
| Dojo 活体示例 | v2.3 文档站闭环。 | 活体 demo 必须跑 Lime fixture / App Server current fixture，不调用正式模型。 |

## v2.0 实施范围

本轮只实施最小可执行内核：

1. 在 `@limecloud/agent-ui-contracts` 新增 `sequenceVerifier.ts`，导出 sequence verifier。
2. verifier 双模式：
   - 批量 `verifyRuntimeEventSequence(events)`，供 fixture / conformance runner。
   - 增量 `createRuntimeSequenceVerifier()` 的 `push(event)` / `finalize()` / `getViolations()`，供未来 runtime client 在分发 listener 前逐事件流式拦截。
3. verifier 至少覆盖以下 violation：
   - `duplicate_event_id`：`id` 去重。
   - `tool_result_without_start` / `tool_failed_without_start`：tool 收口事件无对应 `tool.started`。
   - `tool_started_already_active`：同 `toolCallId` 重复开启。
   - `tool_unclosed_at_turn_end`：turn 终态时仍有未收口的 tool。
   - `action_resolved_without_request`：`action.resolved` 无对应 `action.required`。
   - `action_required_already_active`：同 `actionId` 重复开启。
   - `action_unresolved_at_turn_end`：turn 终态时仍有未解决的 action。
   - `model_unclosed_at_turn_end`：turn 终态时 model 流未收口。
   - `execution_after_turn_terminal`：turn 终态后仍出现同 turn 的 `tool.*` / `action.*` / `model.*` 等执行流事件。
   - `turn_terminal_repeated`：同 turn 重复进入终态。
4. `@limecloud/agent-runtime-projection` 的 `replayAgentUiFixture` 先通过 verifier；有 violation 时 **fail closed**，不把坏流继续投影成 state。
5. contracts / projection 包补定向测试，证明坏流被拦截、好流不被误伤。

### 兼容约束（实施时必须遵守）

verifier 检查的是「违反规则」，**不是「不完整」**。以下情形必须保持合法，否则会误伤现有 fixture：

- 未达 turn 终态的进行中流（例如 `tool-failure`、`hitl-action`、`artifact-evidence`、`subagent-handoff` 这些没有 `turn.completed` 的 fixture），其未闭合 tool / action **不算 violation**，因为流可能仍在进行或只截取片段。
- 收口检查只在 turn 终态事件（`turn.completed` / `turn.failed`）真实出现时触发。
- `snapshot.updated` / `session.*` / `state.*` 等帧外维护事件不算执行流，可在终态后出现，不触发 `execution_after_turn_terminal`。
- `stream-repair` 的 sequence gap 由现有 `collectSequenceIssues` 负责，verifier 不重复判定连续性。

## 非目标

- 本轮不新增 Electron IPC、App Server JSON-RPC 方法或生产 bridge。
- 本轮不把 AG-UI Observable / AbstractAgent API 搬进 Lime。
- v2.0 不把 projector 重写成增量 apply；该债已在 v2.1 完成。
- 本轮不新增 React UI surface，不改 Content Studio 或 Plugin 产品页面。
- v2.0 不实现 JSON Patch 字段级 delta；该 seed 已在 v2.2 完成。
- 本轮不做 npm registry 发布。

## v2.1 实施范围

1. Runtime client pipeline：`subscribeEvents` / `nextEvent` 在分发 listener 前可选接入 verifier，默认 fail closed，测试夹具可显式 collect diagnostics。
2. Projector 增量化：把 `createAgentUiProjector.apply` 的全量重投影改成增量 reducer，消除 O(N²)。
3. App Server event normalization：runtime client 在送入 verifier 前把 `message.delta` / `message.completed` / `turn.final_done` 等 current App Server 事件规整为 Lime 标准 `model.*` / `turn.completed`。
4. Adapter / middleware / versioning 留作后续 v2.x，不算 v2.1 完成判定。

## v2.2 实施范围

1. 设计 `state.delta` 字段级 patch 语义，优先评估 RFC 6902 JSON Patch。
2. 明确 snapshot / delta / hydration / repair cursor 的合并顺序。
3. 输出 JSON Schema，支撑 Rust App Server 侧机械校验。

## v2.5 实施范围

1. `state.delta` 进入 `@limecloud/agent-runtime-projection` 的 `projectAgentUiState()` 与 `createAgentUiProjector.apply()`，batch 与 incremental 共用同一 patch 归并逻辑。
2. patch 失败进入 `hydration.status = "stale"` 并写入 diagnostics，不污染目标 state。
3. `state.delta` 只 patch schema 已知 projection / readModel 子树，不得伪造 runtime fact。

## v2.6 实施范围

1. Rust App Server 在 `RuntimeCore` event 入库前，把 `AgentEvent` 规范化为 Workbench `AgentRuntimeExecutionEvent` 形状，并用 `@limecloud/agent-ui-contracts` checked-in JSON Schema 做机械校验。
2. `state.delta` payload 额外通过 `agent-runtime-state-delta.v0.1.schema.json`，非法 patch fail closed，不写入 session state，也不产生 outbound notification。
3. 契约守卫固定 App Server schema gate，防止后续只保留 TS / 前端下游校验。

## v2.7 实施范围

1. Workbench 文档站新增活体 Demo 矩阵页面，真实执行 `agentUiConformanceFixtures -> validation -> sequence verifier -> replayAgentUiFixture -> projection summary`。
2. 矩阵覆盖 text、tool success/failure、HITL、artifact/evidence、stream repair、Subagents 多事件族，不再只靠 `/subagents` 单点证明闭环。
3. 后续若继续加深，优先补 App Server 真实 fixture backend 与 bad-stream 负向可视化。

## v2.8 实施范围

1. Lime Rust/App Server 在 RuntimeCore event 入库前执行 AgentUI sequence gate，不只依赖 TS runtime client / renderer gateway。
2. gate 与 TS sequence verifier 保持同一核心语义：`tool.started -> tool.result/failed`、`action.required -> action.resolved`、turn terminal 前 active tool/action 必须收口，终态后执行流不得继续污染 state。
3. sequence violation fail closed：不写入 `StoredSession.events`，不提前改变 `AgentTurnStatus` / `AgentSessionStatus`，不发 outbound notification。
4. 保持兼容约束：未达终态的进行中流不算 violation；`message.delta -> turn.final_done` 仍合法；缺少 scope id 的历史兼容事件不被突然误伤。

## v2.9 实施范围

v2.9 补 AG-UI `transformChunks` 背后的承载机制，但不声称已经兼容 AG-UI wire protocol。

1. `@limecloud/agent-runtime-client` 的 event pipeline 支持 adapter / middleware 返回 `0..N` 个 `agentSession/event` notification，并支持 `flush()` / `flushSync()` 让 buffered transform 在流收尾时排空。
2. 根 runtime client 与 `sessionGateway` 的 `dispatchEvent()` / `nextEvent()` 都消费 `pipelineResult.notifications`，fan-out 后的后续事件先进入 pending queue，再读取下一条 transport message。
3. Lime 本体 current event gateway 不再只把 pipeline 当 boolean gate：App Server notification、本地 publish、bridge listener 都消费 pipeline 投影后的 `0..N` payload；未配对 `tool.result` 继续 fail-closed。
4. Plugin current runtime client 工厂透传 runtime-client options，让 adapters / middlewares / verifier mode 能接入 Plugin current 路径。
5. 只落最小兼容 fan-out：明确的 `tool.completed` / `tool.complete` 可展开为 `tool.started + tool.result`。具体 provider chunk family mapping 留给后续，不把临时兼容写成新协议事实源。

## v2.10 实施范围

v2.10 补 AG-UI 仍领先的运行时能力声明和 resume contract，但不新增平行 runtime owner，也不声称已经实现自动 negotiation：

1. `@limecloud/agent-ui-contracts` 新增 `AgentRuntimeCapabilityManifest` / `AgentRuntimeCapabilityEntry` 类型，声明 transport、tools、state.delta、reasoning、multimodal、HITL、subagents、evidence 等支持面。
2. `@limecloud/agent-ui-contracts` 新增 `AgentRuntimeResumeContract` / `AgentRuntimeResumeActionDecision` 类型，固定 open action 的 resume 覆盖规则。
3. contracts 包导出 capability manifest / resume contract JSON Schema constants，并写入 checked-in schema 文件。
4. validation API 机械校验 manifest 必填字段、capability entry 结构，以及 `all-open-actions` / `selected-actions` 模式必须覆盖所有 open action。
5. Lime 本体接入同一合同：App Server `capability/list` response 返回 `runtimeCapabilityManifest`，`agentSession/thread/resume` params 可带 `resumeContract`，RuntimeCore 在启动 queued turn 前校验 schema version、session match 和 open action coverage；前端 `AgentRuntime` current gateway 可读取 manifest，并在 resume 前构造/校验合同。
6. 保持边界：Lime 已有 App Server `capability/list` 和 `resumeAgentSessionThread` 主链继续作为 runtime owner；本轮不新增 App Server method，不实现自动 provider negotiation。

## v2.11 后续

补 projection reconciliation 与外部 transport compatibility 的深水区：

1. RuntimeCore tool lifecycle owner guard：入库前阻断孤立 `tool.args` / `tool.args.delta` / `tool.output.delta`，并要求带 `toolCallId` 的 `permission.denied` / `sandbox.blocked` / `action.required` 只能关联 active tool。2026-06-13 已完成第一刀。
2. RuntimeCore event append batch atomic guard：同一批 runtime events 先完整通过 schema / sequence / tool lifecycle gate，再统一落库和推进 turn/session 状态。2026-06-13 已完成第二刀。
3. RuntimeCore approval gate guard：带 `toolCallId` 的 `action.required` 未收口前阻断同一 tool 输出 / 成功 / 失败；action 取消、过期或拒绝后禁止成功 `tool.result`，只能失败收口。2026-06-13 已完成第三刀。
4. RuntimeCore tool owner adjacency guard：显式 `messageId` / `assistantMessageId` / `itemId` 必须与 `tool.started` 保持同一 owning assistant item；声明 owner 的 tool terminal 缺 owner 或 owner 冲突时 fail closed。2026-06-13 已完成第四刀。
5. RuntimeBackend tool args fact：底层 `ToolStart { arguments }` 必须展开为 `tool.started -> tool.args`，JSON 参数进入 `args`，非 JSON 参数保留为字符串和 `rawArgs`。2026-06-13 已完成第五刀。
6. RuntimeBackend tool failure terminal fact：底层 `ToolEnd { result.success: false }` 必须输出 `tool.failed`，并保留 `toolCallId`、`status=failed`、`failureCategory`、`error` 和 `output`；成功结果继续输出 `tool.result`。2026-06-13 已完成第六刀。
7. RuntimeBackend tool events owner extraction：底层 runtime tool 事件类型映射、tool args 展开、tool terminal payload enrichment 和基础单测统一收敛到 `runtime_backend/tool_events.rs`；中心 `runtime_backend.rs` 只保留执行接线与 coding mirror 编排。2026-06-13 已完成第七刀。
8. ToolRouter / ToolOrchestrator execution slice：仍需把 MCP / ACP / skills / shell / project tools 的 dispatch、approval、sandbox、retry、cancel 收敛到 RuntimeCore 单一执行 owner。
9. message snapshot 与 streaming text 的合并 / 替换规则。
10. partial tool args / JSON args buffer 的归并规则。
11. encrypted reasoning continuity：只保留可展示 summary / encrypted ref，不泄露 raw chain of thought。
12. SSE / protobuf / Accept negotiation 只能作为外部 gateway compatibility 候选；Lime 本体 current transport 仍是 App Server JSON-RPC / Host bridge。

## 完成判定

v2.0 完成必须同时满足：

1. `internal/roadmap/agentworkbench/README.md` 指向本文。
2. contracts 包导出 sequence verifier API（批量 + 增量双模式）。
3. 全部 conformance fixtures 通过 verifier（好流不被误伤）。
4. 至少四个负向测试覆盖：未配对 tool、孤立 tool.result、未配对 action、turn 终态前执行流未收口。
5. projection fixture replay 在 verifier 有 violation 时 fail closed，不继续投影坏流。
6. 定向包测试通过：`@limecloud/agent-ui-contracts` 与 `@limecloud/agent-runtime-projection`。
7. Workbench 文档站补 sequence verifier API 参考，并接入导航。
